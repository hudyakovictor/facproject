## 2. КАК ЗОНЫ ОХВАТЫВАЮТСЯ ДЛЯ КАЖДОГО РАКУРСА

Код `pose_policy.py` (`PosePolicy`) назначает каждой зоне (`A01`...`A20`) роль в зависимости от `yaw_bin`:

```python
# Пример из _build_default():
# Фронтальные зоны (A01-A06, A13-A16, A19-A20):
#   yaw <= |25°|  -> primary    (вес 1.0)
#   |25°| < yaw <= |40°|  -> support  (вес 0.6)
#   |40°| < yaw <= |60°|  -> limited  (вес 0.25)
#   |yaw| > |60°|  -> exclude   (вес 0.0)
#
# Левые зоны (A07, A09, A11, A17):
#   10° <= yaw <= 40°  -> primary
#   -10° <= yaw < 10°  -> support
#   -25° <= yaw < -10° -> limited
#
# Правые зоны (A08, A10, A12, A18):
#   -40° <= yaw <= -10°  -> primary
#   -10° < yaw <= 10°   -> support
#   10° < yaw <= 25°    -> limited
```

## 3. ПРОВЕРКА ПРОЕКЦИИ НА АТЛАС (`projection.py`)

### 3.1 Что происходит при проекции фото на атлас

```
Фото (bgr) -> skin_mask (face_mask.npz) -> crop -> rasterize_surface
                                           -> normals, depth, visibility
                                           -> surface_vertices, triangle_surface_areas
                                           -> project_atlas (atlas.A, atlas.S, atlas.W)
                                           -> zone_id_map (A20), zone_mask_photo
                                           -> skin_zone_projection.npz
```

### 3.2 Физика проекции (v4 fix)

Новая функция `projected_density_map` в `projection.py`:

```python
# density = screen_pixel_count / triangle_surface_area
projected_density[tid==t_id] = cnt / sa
```

Это позволяет корректно сравнивать плотность пикселей на единицу поверхности для разных фото с разным масштабом или расстоянием.

## 4. ПРОБЛЕМА НАЛОЖЕНИЯ ДВУХ ФОТО ОДНОГО РАКУРСА С РАЗНЫМ НАКЛОНОМ ГОЛОВЫ

### 4.1 Текущая ситуация в коде

Код `pair_comparison.py` (`compare_packages`) использует `common_surface()` для вычисления общей наблюдаемой поверхности зон (`coverage_sym`). Но:

1. **Нет геометрической нормализации (`geometric normalization`)** перед сравнением морщин или текстуры.
2. **Нет механизма `ideal overlay`** — код не проверяет, совпадают ли форма и положение морщин на уровне пикселей после нормализации.
3. **`pose_policy.is_compatible()`** проверяет только совместимость зон (`primary`/`support`/`limited`/`exclude`), но не проверяет, совпадают ли позиции морщин.

### 4.2 Что нужно для «идеального наложения»

Для двух фото с одинаковым `pose_bin` (например, оба `frontal`) но разным `yaw` в пределах диапазона (например, -8° и +5°):

1. **Нормализовать UV-координаты** — применить `mesh_alignment_residual` или аналогичную коррекцию перспективы.
2. **Выровнять морщинные маски (`W14`)** через геометрическую трансформацию (`geometric_transform`).
3. **Сравнить наложенные маски** через `SSIM` или `pixel-level overlap`.

**ВЫВОД:** Код готов к проекции, но **НЕ ГОТОВ** к идеальному наложению для сравнения морщин одного человека с разным наклоном головы.
